# Power BI Diagnostics

> Research date: April 2026

## Performance Analysis

### Performance Analyzer (Built-in)

Power BI Desktop's built-in tool for identifying slow visuals and queries.

**How to Use:**

1. Open report in Power BI Desktop
2. View tab > Performance Analyzer > Start Recording
3. Interact with report (refresh visuals, change slicers)
4. Review results sorted by duration

**Metrics Captured Per Visual:**

| Metric | What It Measures |
|---|---|
| **DAX Query** | Time to execute the DAX query generated by the visual |
| **Visual Display** | Time to render the visual in the browser/canvas |
| **Other** | Time for other operations (e.g., preparing queries) |
| **Total** | Combined time for the visual |

**Key Actions:**

- Copy the DAX query from any visual and paste into DAX Studio for deeper analysis
- Export all results to a JSON file for offline analysis
- DAX Studio can import Performance Analyzer data directly for sorting, filtering, and optimization

### DAX Studio Query Analysis

DAX Studio provides deeper analysis than Performance Analyzer.

**Server Timings:**

1. Connect to local model or XMLA endpoint
2. Enable Server Timings (bottom toolbar)
3. Run the DAX query
4. Review breakdown:
   - **Total**: Overall query time
   - **SE (Storage Engine)**: Time in VertiPaq; highly optimized, multithreaded
   - **FE (Formula Engine)**: Time in formula processing; single-threaded bottleneck
   - **SE Queries**: Number of storage engine queries generated
   - **SE Cache**: Whether cached results were used

**Interpreting Results:**

- High FE time = DAX formula optimization needed (restructure measures, use variables, avoid iterators)
- High SE time = Data model optimization needed (reduce cardinality, remove columns, improve relationships)
- Many SE Queries = Query plan is suboptimal; consider restructuring DAX
- SE Cache hits = Good; repeated subqueries are being cached

**Query Plan Analysis:**

- View logical and physical query plans
- Identify materialization operations (data moved from SE to FE)
- Large materializations indicate performance bottlenecks

### VertiPaq Analyzer

Analyzes the in-memory data model structure and identifies optimization opportunities.

**Access Methods:**

- DAX Studio: Advanced tab > View Metrics
- Tabular Editor 3: Built-in VertiPaq Analyzer
- Export as VPAX file for sharing/archival

**Key Metrics to Review:**

| Metric | What to Look For |
|---|---|
| **Table Size** | Identify largest tables consuming memory |
| **Column Cardinality** | High-cardinality columns compress poorly; consider removing or bucketing |
| **Column Size** | Columns taking disproportionate space relative to value |
| **Dictionary Size** | Large dictionaries indicate high cardinality or long strings |
| **Relationship Size** | Large relationships may indicate modeling issues |
| **Referential Integrity Violations** | Missing keys between related tables; performance impact |
| **Encoding Type** | Value vs Hash encoding; VertiPaq chooses automatically |

**Common Findings and Fixes:**

- Remove unused columns (especially high-cardinality text columns)
- Remove auto date/time tables if custom date table exists
- Split high-cardinality columns (e.g., full datetime to date + time)
- Address referential integrity violations (orphan keys in fact tables)

## Common Issues and Resolution

### Slow Reports

**Diagnosis Steps:**

1. Run Performance Analyzer to identify slowest visuals
2. Copy DAX queries to DAX Studio for Server Timings analysis
3. Check VertiPaq Analyzer for model bloat
4. Review report design (visual count, slicer count)

**Common Causes and Fixes:**

| Cause | Symptom | Fix |
|---|---|---|
| Too many visuals per page | All visuals slow | Reduce to 8 or fewer; use drillthrough pages |
| Complex DAX measures | High FE time in DAX Studio | Use variables, avoid nested iterators, use native aggregations |
| High-cardinality columns | Large model size in VertiPaq Analyzer | Remove or bucket unnecessary columns |
| Excessive slicers | Slow initial load and interactions | Consolidate slicers; use filter pane instead |
| Bidirectional relationships | Unpredictable slow queries | Switch to single-direction; use DAX CROSSFILTER |
| Large tables in Import mode | High memory, slow refresh | Implement incremental refresh, aggregations |
| Missing relationships | Incorrect or slow results | Verify star schema with proper relationships |

### Data Refresh Failures

**Diagnosis Steps:**

1. Check Refresh History: Semantic model settings > Refresh > Refresh history
2. Review error messages (status, duration, error details)
3. Test data source connectivity from gateway machine
4. Verify credentials in Data source credentials settings

**Common Causes:**

| Error | Cause | Fix |
|---|---|---|
| Credential error | Expired OAuth token or changed password | Re-enter credentials in Data source credentials |
| OAuth token expired during refresh | Refresh takes longer than 1-hour token lifetime | Optimize queries to reduce refresh time; use incremental refresh |
| Gateway not reachable | Gateway offline or outdated | Restart gateway service; update to latest version |
| Data source not found | Connection string changed or source offline | Update connection in semantic model settings |
| Privacy level conflict | "Information needed to combine data" error | Set privacy levels explicitly in data source settings |
| Memory exceeded | Model too large for capacity | Reduce model size; upgrade capacity; use incremental refresh |
| Timeout | Query exceeds timeout threshold | Optimize Power Query; implement query folding; use incremental refresh |

**Prevention:**

- Set up email alerts for refresh failures
- Monitor refresh history regularly
- Use incremental refresh for large datasets
- Keep gateway software current
- Document data source dependencies

### Gateway Connectivity Issues

**On-Premises Data Gateway:**

The gateway bridges on-premises data sources with the Power BI Service. It is a Windows service that must run continuously.

**Troubleshooting Steps:**

1. **Verify gateway version**: Always update to latest; many issues resolved in monthly updates
2. **Check Windows service**: Ensure "On-premises data gateway" service is running
3. **Test local connectivity**: Open Power BI Desktop on the gateway machine and test the data source connection
4. **Review gateway logs**: Located in `%localappdata%\Microsoft\On-premises data gateway\`
5. **Check firewall**: Gateway needs outbound HTTPS (port 443) to Azure Service Bus
6. **Verify credentials**: Re-enter data source credentials in Power BI Service gateway settings

**Gateway Cluster Considerations:**

- **Concurrency limit**: Default 30 concurrent queries per gateway node
- **Load balancing**: Add gateway nodes to a cluster for higher throughput
- **Failover**: Multiple nodes provide redundancy if one node goes down

**Personal Gateway:**

- Simpler setup for individual use
- Only supports Import mode (no DirectQuery)
- Runs under the user's Windows account
- Not recommended for production/team scenarios

**Fabric Integration:**

- Fabric VNet data gateways provide managed gateway infrastructure
- Eliminates need to install and maintain on-premises gateway software
- Available for Fabric capacity workspaces

## Capacity Management

### Microsoft Fabric Capacity Metrics App

The primary tool for monitoring capacity health and performance.

**Key Metrics:**

| Metric | Description | Threshold |
|---|---|---|
| **CU Utilization %** | CPU usage relative to capacity v-cores | Warning at 80%+ sustained |
| **Interactive Operations** | User-triggered queries (report views, slicer changes) | Monitor for throttling events |
| **Background Operations** | Scheduled refreshes, dataflow runs | Stagger to avoid peaks |
| **Throttling Events** | Operations delayed due to capacity saturation | Target: zero during business hours |
| **Overload Events** | Operations rejected due to sustained overload | Requires immediate action |

### Throttling Behavior

When capacity is overloaded:
1. **Interactive throttling**: User queries are delayed (not rejected)
2. **Background throttling**: Scheduled refreshes are deferred
3. **Autoscale activation** (if configured): Additional v-cores added
4. **Rejection**: Sustained overload leads to operation rejection

### Autoscale

- Safety net for capacity spikes, not a primary capacity strategy
- Adds v-cores in 30-second increments when interactive throttling occurs
- Maximum number of autoscale v-cores configured by admin
- Billed separately from base capacity

**When to Upgrade Base SKU Instead:**

- Autoscale activates more than 3 times per week
- Sustained CU utilization above 80% during business hours
- Interactive throttling events occur more than 3 times per week
- Autoscale activates daily (likely cheaper to upgrade SKU)

### Capacity Optimization Strategies

1. **Stagger refresh schedules**: Spread across 2-hour windows instead of all at midnight
2. **Reduce dataset sizes**: Remove unused columns and tables
3. **Implement incremental refresh**: Only refresh changed data
4. **Limit visuals per page**: 8-12 maximum
5. **Use aggregations**: Serve common queries from pre-aggregated tables
6. **Separate dev/test**: Move non-production workloads to a different capacity
7. **Monitor with Capacity Metrics App**: Weekly review of utilization patterns

## DAX Debugging

### EVALUATE in DAX Studio

Use EVALUATE to test DAX expressions outside of reports.

```dax
-- Test a measure in isolation
EVALUATE
SUMMARIZECOLUMNS(
    'Date'[Year],
    'Date'[Month],
    "Total Sales", [Total Sales],
    "YTD Sales", [YTD Sales]
)

-- Filter to specific context
EVALUATE
CALCULATETABLE(
    SUMMARIZECOLUMNS(
        'Product'[Category],
        "Revenue", [Revenue]
    ),
    'Date'[Year] = 2025
)
```

### Step-by-Step Measure Analysis

For complex measures, decompose into components:

1. **Isolate the measure**: Run the measure standalone with EVALUATE
2. **Add filter context**: Apply filters incrementally to understand behavior
3. **Break into parts**: Use VAR to decompose measure into intermediate values
4. **Examine each part**: Return each variable as a separate column to verify
5. **Check Server Timings**: Identify which part is the bottleneck

```dax
-- Debugging a complex measure
DEFINE
    VAR Revenue = [Total Revenue]
    VAR Cost = [Total Cost]
    VAR Margin = Revenue - Cost
    VAR MarginPct = DIVIDE(Margin, Revenue)
EVALUATE
ROW(
    "Revenue", Revenue,
    "Cost", Cost,
    "Margin", Margin,
    "Margin %", MarginPct
)
```

### Common DAX Debugging Scenarios

| Issue | Diagnostic Approach |
|---|---|
| Measure returns BLANK | Check filter context; verify relationships; test with EVALUATE |
| Measure returns wrong total | Check for context transition issues; test with ALL/REMOVEFILTERS |
| Slow measure | Server Timings in DAX Studio; check SE vs FE split |
| Circular dependency error | Review calculated column/table dependencies; restructure logic |
| Unexpected duplicates | Check relationship cardinality; verify key uniqueness in dimensions |

## Data Refresh Diagnostics

### Refresh Types

| Type | Trigger | Monitoring |
|---|---|---|
| **Scheduled Refresh** | Time-based (up to 8/day Pro, 48/day PPU) | Refresh history in settings |
| **On-Demand Refresh** | Manual trigger in Service | Refresh history |
| **Incremental Refresh** | Partition-based, only changed data | XMLA endpoint; Refresh history |
| **Real-Time (Hybrid)** | DirectQuery partition for latest data | DirectQuery diagnostics |
| **API Refresh** | REST API trigger (CI/CD pipelines) | API response + Refresh history |

### Refresh Failure Analysis

**Step 1: Check Refresh History**
- Navigate to semantic model > Settings > Refresh > Refresh History
- Review status, duration, and error messages

**Step 2: Categorize the Error**
- **Credential errors**: Re-authenticate data sources
- **Connectivity errors**: Check gateway status, network, firewall
- **Data errors**: Source schema changes, null violations, type mismatches
- **Timeout errors**: Optimize queries, implement incremental refresh
- **Capacity errors**: Model too large, insufficient memory

**Step 3: Test in Desktop**
- Open the .pbix file on the gateway machine
- Attempt refresh in Desktop to reproduce the error
- Check Power Query steps for errors

**Step 4: Review Gateway Logs** (if applicable)
- Location: `%localappdata%\Microsoft\On-premises data gateway\`
- Look for connection failures, authentication errors, timeout messages

### Automatic Retry and Deactivation

- Power BI retries after certain transient errors
- After 4 consecutive scheduled refresh failures, the schedule is automatically deactivated
- Must resolve the issue and manually re-enable the schedule
- Set up email notifications for refresh failures to catch issues early

### Credential Management

- OAuth tokens expire approximately every 1 hour
- If refresh exceeds token lifetime, credential error occurs even with valid credentials
- For long-running refreshes: optimize query time, implement incremental refresh
- Force credential refresh: Navigate to `https://app.powerbi.com?alwaysPromptForContentProviderCreds=true`
- Service principal authentication does not have OAuth expiry issues

## Monitoring and Alerting

### Built-in Monitoring

| Tool | Purpose | Access |
|---|---|---|
| **Refresh History** | Per-dataset refresh status and errors | Semantic model settings |
| **Usage Metrics** | Report views, unique viewers, performance | Report/workspace context menu |
| **Capacity Metrics App** | CU utilization, throttling, background ops | Premium/Fabric capacity admin |
| **Admin Portal** | Tenant-wide usage, audit logs, governance | Power BI admin role |
| **Power BI Activity Log** | Detailed audit events via REST API | Admin API or PowerShell |

### External Monitoring

| Tool | Purpose |
|---|---|
| **DAX Studio** | Query performance analysis, VertiPaq metrics |
| **Tabular Editor** | Best Practice Analyzer, model documentation |
| **Azure Monitor** | Capacity-level diagnostics and alerting |
| **Power BI REST API** | Programmatic monitoring and automation |
| **Power Automate** | Automated alerting on refresh failures or capacity events |

### Recommended Alerting Setup

1. **Email subscriptions** for refresh failures (built-in)
2. **Power Automate flows** for custom alerting (Teams, Slack, email)
3. **Capacity Metrics App** alerts for throttling/overload
4. **Azure Monitor** for capacity diagnostics at scale
5. **Scheduled Power BI Activity Log** exports for compliance auditing
