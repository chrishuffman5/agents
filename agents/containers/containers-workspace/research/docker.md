# Docker Engine Deep Dive
*Research date: April 2026*

---

## Architecture Overview

Docker Engine uses a layered, modular architecture built around open standards (OCI). The request flow is:

```
Docker CLI → Docker Daemon (dockerd) → containerd → containerd-shim-runc-v2 → runc → Linux Kernel
```

### Docker Daemon (dockerd)

The daemon listens for Docker API requests via a Unix socket (`/var/run/docker.sock`), TCP socket, or named pipe (Windows). It manages:
- Image builds (delegated to BuildKit)
- Container lifecycle
- Volume and network management
- Delegating execution to containerd

As of Docker Engine v29, the daemon no longer manages its own image store by default — that responsibility moved to containerd.

### containerd

containerd is the industry-standard container runtime, a CNCF graduated project. Docker Engine bundles containerd as its execution backend. containerd handles:
- Image pull, push, and storage (content store)
- Container execution and supervision via shims
- Namespace isolation (containerd namespaces separate Docker from other runtimes)
- Snapshotters (overlayfs by default)
- CRI implementation for Kubernetes

Docker Engine v29 ships containerd static binaries at v2.2.2.

### runc

runc is the OCI reference implementation for container creation. It:
- Reads the OCI runtime spec (`config.json`) generated by containerd
- Sets up Linux namespaces (pid, net, mnt, uts, ipc, user)
- Configures cgroups (v1 or v2) for resource limits
- Applies seccomp filters and capabilities
- Executes the container process

`containerd-shim-runc-v2` is the shim that manages runc processes, keeping containers running if containerd restarts.

### OCI Runtime Specification

The Open Container Initiative (OCI) defines two specs:
- **Runtime Spec**: how to run a container (namespaces, mounts, process config)
- **Image Spec**: container image format (layers as tarballs + JSON manifests)
- **Distribution Spec**: how registries distribute images

runc is the reference OCI runtime; alternatives include crun (C, faster), gVisor/runsc (sandboxed), kata-containers (VM-based), and youki (Rust).

## Docker Engine Versions

| Version | Release     | Key Changes |
|---------|-------------|-------------|
| 25.0    | Jan 2024    | BuildKit default, containerd image store opt-in |
| 26.0    | Mar 2024    | IPv6 dual-stack improvements, compose GA |
| 27.0    | Jun 2024    | Performance improvements, CVE patches |
| 28.0    | Dec 2024    | nftables experimental, API minimum raised |
| 29.0    | Feb 2026    | containerd image store default, nftables experimental stable |
| 29.3.1  | Mar 2026    | Latest stable; HTTP keep-alive for registry, containerd 2.2.2 |

### Docker Engine v29 Key Changes

- **containerd image store is now the default** for all new installations, replacing Docker's legacy graphdriver backend
- **Minimum API version raised to 1.44** — clients using older API versions must upgrade
- **Experimental nftables support**: Docker can now generate nftables rules directly instead of routing through iptables-nft translation layer
- **Image pull/push performance**: HTTP keep-alive for registry connections eliminates redundant TCP/TLS handshakes
- **Security fixes**: plugin installation hardening, Git URL validation in BuildKit, untrusted frontend protection

### Docker Desktop vs Docker Engine

| Feature              | Docker Desktop           | Docker Engine (Linux) |
|---------------------|--------------------------|----------------------|
| Platform            | macOS, Windows, Linux    | Linux only           |
| License             | Paid for large orgs      | Apache 2.0 (free)    |
| VM isolation        | Yes (HyperKit/WSL2/VZ)  | No (native)          |
| GUI dashboard       | Yes                      | No                   |
| Dev environments    | Yes                      | No                   |
| Kubernetes (built-in)| Yes (optional)          | No                   |
| Extensions          | Yes                      | No                   |
| rootless default    | Yes (on macOS)           | Optional             |

## Docker CLI Essential Commands

### Container Lifecycle
```bash
docker run -d --name myapp -p 8080:80 --restart unless-stopped nginx
docker run -it --rm alpine sh                    # interactive, auto-remove
docker run --cpus 1.5 --memory 512m myimage      # resource limits
docker exec -it myapp bash                       # exec into running container
docker logs -f --tail 100 myapp                  # follow logs
docker inspect myapp                             # full JSON metadata
docker stats                                     # live resource usage
docker stop myapp && docker rm myapp
docker container prune                           # remove all stopped
```

### Image Management
```bash
docker pull nginx:1.27-alpine
docker images --filter dangling=true
docker rmi $(docker images -q --filter dangling=true)
docker image prune -a --filter "until=24h"
docker save myimage | gzip > myimage.tar.gz
docker load < myimage.tar.gz
docker tag myimage registry.example.com/myimage:v1.0
docker push registry.example.com/myimage:v1.0
```

### System
```bash
docker system df                                 # disk usage
docker system prune -a --volumes                # full cleanup
docker info                                      # daemon info
docker version                                   # client + server versions
docker context ls                                # list contexts
docker context use remote-server                 # switch context
```

## Dockerfile Best Practices

### Multi-Stage Builds

Multi-stage builds are the primary mechanism for producing minimal production images:

```dockerfile
# Build stage
FROM golang:1.23-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /app/server .

# Final stage — scratch or distroless
FROM gcr.io/distroless/static-debian12:nonroot
WORKDIR /app
COPY --from=builder /app/server .
EXPOSE 8080
ENTRYPOINT ["/app/server"]
```

Benefits: build tools not included in final image, smaller attack surface, better layer caching.

### Layer Caching

Layers are cached by content hash. Optimize by:
1. Put rarely-changing instructions first (OS packages, dependency install)
2. Copy dependency manifests before source code
3. Use `--mount=type=cache` in BuildKit for package managers

```dockerfile
# Good: package files copied first, source after
COPY package.json package-lock.json ./
RUN npm ci --production
COPY src/ ./src/

# Bad: COPY . . invalidates cache on every source change
COPY . .
RUN npm ci
```

### .dockerignore

Always include `.dockerignore` to prevent leaking secrets and reducing build context size:

```
.git
node_modules
*.log
.env
.env.*
dist
coverage
__pycache__
*.pyc
.DS_Store
Thumbs.db
*.md
tests/
docs/
```

### Non-Root Users

```dockerfile
# Create a dedicated user
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
USER appuser

# Or use numeric UID (works better with Kubernetes runAsUser)
RUN adduser -u 10001 -D appuser
USER 10001
```

### HEALTHCHECK

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
```

For services without curl:
```dockerfile
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:8080/health || exit 1
```

### Other Best Practices
- Use specific image tags, not `latest` in production
- Prefer `COPY` over `ADD` (predictable behavior)
- Use `ENTRYPOINT` for the command, `CMD` for default arguments
- Combine `RUN` commands with `&&` to minimize layers
- Use `--no-install-recommends` for apt installs
- Pin package versions for reproducibility

## Docker Compose v2

Docker Compose v2 is a Go rewrite integrated as a Docker CLI plugin (`docker compose`). The legacy Python-based `docker-compose` (v1) is end-of-life. The Compose Specification v5.0.0 was released in December 2025 and removed the internal builder in favor of Docker Bake.

### compose.yaml Structure

```yaml
# compose.yaml (or docker-compose.yaml)
name: myapp

services:
  api:
    build:
      context: ./api
      dockerfile: Dockerfile
      target: production
    image: myapp/api:${TAG:-latest}
    ports:
      - "8080:8080"
    environment:
      - DATABASE_URL=postgresql://db:5432/myapp
    env_file:
      - .env.local
    depends_on:
      db:
        condition: service_healthy
    networks:
      - backend
    volumes:
      - api-logs:/app/logs
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    volumes:
      - postgres-data:/var/lib/postgresql/data
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d myapp"]
      interval: 10s
      timeout: 5s
      retries: 5
    secrets:
      - db_password

networks:
  backend:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/16

volumes:
  postgres-data:
    driver: local
  api-logs:

secrets:
  db_password:
    file: ./secrets/db_password.txt
```

### Profiles

Profiles enable conditional service activation:

```yaml
services:
  app:
    image: myapp
    # always started

  debug-proxy:
    image: mitmproxy/mitmproxy
    profiles: [debug]

  load-test:
    image: k6
    profiles: [testing]
```

```bash
docker compose --profile debug up        # start app + debug-proxy
docker compose --profile testing run load-test
```

### Watch Mode

Watch mode (stable since Compose v2.22.0) monitors files and auto-syncs or rebuilds:

```yaml
services:
  app:
    build: .
    develop:
      watch:
        - action: sync
          path: ./src
          target: /app/src
          ignore:
            - node_modules/
        - action: rebuild
          path: package.json
        - action: sync+restart
          path: ./config
          target: /app/config
```

```bash
docker compose watch       # start with file watching
```

- `sync`: copies files into running container (no restart) — ideal for interpreted languages
- `rebuild`: triggers full `docker compose build && up`
- `sync+restart`: syncs then restarts the service

## Docker Build (BuildKit)

BuildKit is the default build engine since Docker 23.0. It provides concurrent, cache-efficient builds.

### buildx

`docker buildx` is the BuildKit CLI frontend:

```bash
# Create a builder instance
docker buildx create --name mybuilder --use --bootstrap

# Multi-platform build and push
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag registry.example.com/myapp:v1.0 \
  --push \
  .

# Build with provenance and SBOM (supply chain security)
docker buildx build \
  --provenance=true \
  --sbom=true \
  --platform linux/amd64,linux/arm64 \
  --push \
  --tag myapp:v1.0 .
```

### Cache Export/Import

```bash
# Export cache to registry
docker buildx build \
  --cache-to type=registry,ref=registry.example.com/myapp:cache,mode=max \
  --cache-from type=registry,ref=registry.example.com/myapp:cache \
  --push -t registry.example.com/myapp:latest .

# GitHub Actions cache
docker buildx build \
  --cache-to type=gha,mode=max \
  --cache-from type=gha \
  --push -t myapp:latest .

# Local cache
docker buildx build \
  --cache-to type=local,dest=/tmp/buildcache,mode=max \
  --cache-from type=local,src=/tmp/buildcache \
  .
```

Cache backends: `registry`, `local`, `gha` (GitHub Actions), `s3`, `azblob`, `inline`.
`mode=max` caches all intermediate layers; `mode=min` (default) caches only final layers.

### Build Secrets

```bash
# Pass a secret from a file (never baked into layers)
docker buildx build \
  --secret id=npmrc,src=$HOME/.npmrc \
  .

# In Dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install
```

### Build Cache Mounts

```dockerfile
# Cache apt packages between builds
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,sharing=locked \
    apt-get update && apt-get install -y build-essential

# Cache Go module downloads
RUN --mount=type=cache,target=/root/.cache/go-build \
    --mount=type=cache,target=/go/pkg/mod \
    go build -o /app/server .
```

### Docker Bake (HCL)

```hcl
# docker-bake.hcl
group "default" {
  targets = ["api", "worker"]
}

target "api" {
  context = "./api"
  dockerfile = "Dockerfile"
  platforms = ["linux/amd64", "linux/arm64"]
  tags = ["registry.example.com/api:latest"]
  cache-from = ["type=registry,ref=registry.example.com/api:cache"]
  cache-to = ["type=registry,ref=registry.example.com/api:cache,mode=max"]
}

target "worker" {
  context = "./worker"
  tags = ["registry.example.com/worker:latest"]
}
```

```bash
docker buildx bake --push
```

## Docker Networking

### Network Drivers

| Driver   | Scope  | Use Case |
|----------|--------|----------|
| bridge   | local  | Default; containers on same host communicate |
| host     | local  | Remove network isolation, use host network stack |
| overlay  | swarm  | Multi-host communication (Docker Swarm / Swarm Mode) |
| macvlan  | local  | Container gets its own MAC and IP on physical LAN |
| ipvlan   | local  | Like macvlan but shares MAC; L2 or L3 modes |
| none     | local  | No networking |

### Bridge Networking

```bash
# Create custom bridge (better than default docker0)
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/16 \
  --gateway 172.20.0.1 \
  --ip-range 172.20.5.0/24 \
  --opt com.docker.network.bridge.name=br-myapp \
  myapp-net

docker run -d --name api --network myapp-net myapp/api
docker run -d --name db --network myapp-net postgres:16
# api container can reach db by hostname 'db'
```

### IPAM (IP Address Management)

Docker uses its internal IPAM by default. Custom IPAM plugins can integrate with external IPAM systems (Infoblox, etc.):

```yaml
networks:
  mynet:
    ipam:
      driver: default
      config:
        - subnet: 192.168.100.0/24
          gateway: 192.168.100.1
          ip_range: 192.168.100.128/25
```

### Overlay (Swarm)

```bash
docker network create -d overlay --attachable myoverlay
```

Overlay uses VXLAN encapsulation (UDP port 4789) to bridge container traffic across hosts.

### Macvlan

Each container gets a unique MAC address and IP on the physical LAN. Requires promiscuous mode on the host NIC. Not supported in rootless mode.

```bash
docker network create -d macvlan \
  --subnet=192.168.1.0/24 \
  --gateway=192.168.1.1 \
  -o parent=eth0 \
  macvlan-net
```

### nftables (Docker Engine v29, Experimental)

Docker Engine 29 can generate nftables rules directly rather than routing through iptables-nft translation. Enable via daemon config:

```json
{
  "iptables": true,
  "experimental": true,
  "features": { "containerd-snapshotter": true }
}
```

## Docker Volumes and Storage Drivers

### Volume Types

```bash
# Named volume (managed by Docker)
docker volume create mydata
docker run -v mydata:/app/data myimage

# Bind mount (host path)
docker run -v /host/path:/container/path myimage

# tmpfs (in-memory, not persisted)
docker run --tmpfs /tmp:rw,size=100m myimage

# Volume with options
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/exports/data \
  nfs-volume
```

### Storage Drivers

| Driver    | Kernel Support | Performance | Notes |
|-----------|---------------|-------------|-------|
| overlay2  | 4.0+          | Good        | Default; requires ftype=1 on xfs |
| fuse-overlayfs | any     | Moderate    | Required for rootless on older kernels |
| btrfs     | btrfs FS      | Good        | Native snapshotting |
| zfs       | ZFS FS        | Good        | Enterprise features |
| devicemapper | legacy    | Poor        | Deprecated, removed in Docker v29 |
| aufs      | legacy        | Poor        | Removed in Docker v29 / containerd 2.0 |

With the containerd image store (default in v29), storage is managed by containerd's snapshotter (overlayfs by default).

## Docker Security

### Rootless Mode

Run Docker daemon as non-root user. Eliminates risk of container escape to root:

```bash
# Install rootless Docker
dockerd-rootless-setuptool.sh install

# Run as normal user
export DOCKER_HOST=unix://$XDG_RUNTIME_DIR/docker.sock
docker run hello-world
```

Limitations in rootless:
- No macvlan/ipvlan network drivers
- No AppArmor (by default)
- Port binding below 1024 requires `net.ipv4.ip_unprivileged_port_start=0`

### User Namespaces

Map container UIDs to unprivileged host UIDs. Daemon-level configuration:

```json
// /etc/docker/daemon.json
{
  "userns-remap": "default"
}
```

`default` maps to `dockremap` user/group. Custom: `"userns-remap": "myuser:mygroup"`.

### seccomp

Docker applies a default seccomp profile that blocks ~44 of 300+ syscalls. Custom profiles:

```bash
docker run --security-opt seccomp=/path/to/profile.json myimage

# Disable seccomp (not recommended)
docker run --security-opt seccomp=unconfined myimage
```

### AppArmor

Docker automatically applies the `docker-default` AppArmor profile on systems where AppArmor is enabled:

```bash
docker run --security-opt apparmor=docker-default myimage
docker run --security-opt apparmor=unconfined myimage    # disable
docker run --security-opt apparmor=my-profile myimage    # custom
```

### Read-Only Containers

```bash
docker run --read-only \
  --tmpfs /tmp \
  --tmpfs /var/run \
  myimage
```

### Capability Dropping

```bash
# Drop all capabilities, add only what's needed
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE myimage
```

### Docker Scout (Vulnerability Scanning)

Docker Scout integrates into `docker buildx build` pipeline for supply chain security:

```bash
# Scan an image for CVEs
docker scout cves myimage:latest

# Compare two images
docker scout compare myimage:v1 myimage:v2

# Quick recommendations
docker scout recommendations myimage:latest

# SBOM (Software Bill of Materials)
docker scout sbom myimage:latest

# Integrate with CI/CD
docker scout quickview --exit-code myimage:latest
```

Docker Scout pulls from VEX (Vulnerability Exploitability eXchange) documents and CVSS scoring.

## Registry

### Docker Hub

- Default registry: `docker.io` (or `registry-1.docker.io`)
- Rate limits (unauthenticated): 100 pulls/6h per IP
- Rate limits (free account): 200 pulls/6h
- Docker Hub Pro/Teams: no rate limits
- Official images, verified publisher images, Docker-sponsored OSS

```bash
docker login
docker pull nginx:1.27-alpine
docker tag myapp:latest dockerhub-username/myapp:latest
docker push dockerhub-username/myapp:latest
```

### Private Registries

```bash
# Login to private registry
docker login registry.example.com

# Configure insecure registry (daemon.json)
{
  "insecure-registries": ["registry.internal:5000"]
}

# Pull from private registry
docker pull registry.example.com/myorg/myapp:v1.0
```

### Harbor

Harbor is the CNCF-graduated enterprise container registry:
- **Security**: built-in vulnerability scanning (Trivy), image signing (cosign/Notary v2)
- **RBAC**: projects, users, robot accounts, LDAP/OIDC integration
- **Replication**: replicate images to/from other registries (Docker Hub, ECR, ACR, GCR)
- **Proxy cache**: cache pulls from Docker Hub to avoid rate limits
- **Webhooks**: trigger CI/CD pipelines on push events
- **Helm chart hosting**: Harbor OCI registry serves Helm charts natively
- **p2p distribution**: Dragonfly or Kraken integration for large-scale distribution
