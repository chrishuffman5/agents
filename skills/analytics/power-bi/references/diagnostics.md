# Power BI Diagnostics Reference

## Performance Analyzer (Built-in Desktop Tool)

### How to Use

1. Open report in Power BI Desktop
2. View tab > Performance Analyzer > Start Recording
3. Interact with report (refresh visuals, change slicers, navigate pages)
4. Review results sorted by duration

### Metrics Per Visual

| Metric | What It Measures |
|---|---|
| **DAX Query** | Time to execute the DAX query generated by the visual |
| **Visual Display** | Time to render the visual in the browser/canvas |
| **Other** | Time for other operations (preparing queries, waiting) |
| **Total** | Combined time for the visual |

### Key Actions

- Copy the DAX query from any slow visual and paste into DAX Studio for deeper analysis
- Export all results to a JSON file; DAX Studio can import Performance Analyzer data directly
- Focus on visuals with DAX Query time > 1 second -- these are optimization candidates
- If Visual Display time dominates, the issue is rendering (too many data points, complex visual type)

## DAX Studio Query Analysis

DAX Studio provides deeper performance analysis than the built-in Performance Analyzer.

### Server Timings

1. Connect to local model (Desktop) or XMLA endpoint (Service)
2. Enable Server Timings (bottom toolbar)
3. Run the DAX query
4. Review the breakdown:

| Metric | What It Means | Optimization Target |
|---|---|---|
| **Total** | Overall query time | < 1 second for interactive visuals |
| **SE (Storage Engine)** | Time scanning VertiPaq data; multithreaded | Should be >90% of total time |
| **FE (Formula Engine)** | Time in DAX evaluation; single-threaded | Should be <10% of total time |
| **SE Queries** | Number of storage engine queries generated | Fewer is better; many SE queries indicate suboptimal query plan |
| **SE Cache** | Whether cached SE results were reused | Cache hits are fast; misses indicate unique query patterns |

### Interpreting Results

| Finding | Root Cause | Fix |
|---|---|---|
| High FE time (>50% of total) | Complex DAX; iterators on large tables; nested CALCULATE | Restructure measures; use variables; replace iterators with native aggregations |
| High SE time | Large table scans; high-cardinality columns; missing relationships | Reduce model size; remove unnecessary columns; check VertiPaq Analyzer |
| Many SE Queries (>20) | Suboptimal query plan; complex joins | Simplify measure; reduce relationship complexity |
| Large materializations | Data moved from SE to FE for processing | Reduce iterator scope; pre-aggregate in Power Query |

### Query Plan Analysis

- View logical and physical query plans in DAX Studio
- Identify materialization operations (data transferred from SE to FE)
- Large materializations (millions of rows) indicate performance bottlenecks
- CallbackDataID operations in SE queries indicate context transition overhead

## VertiPaq Analyzer

Analyzes the in-memory data model structure to identify optimization opportunities.

### Access Methods

- **DAX Studio**: Advanced tab > View Metrics
- **Tabular Editor 3**: Built-in VertiPaq Analyzer
- **VPAX export**: Portable file for sharing model metadata without actual data

### Key Metrics

| Metric | What to Look For | Action |
|---|---|---|
| **Table Size** | Largest tables consuming memory | Focus optimization on the biggest tables first |
| **Column Cardinality** | High-cardinality columns compress poorly | Remove, bucket, or move to dimension tables |
| **Column Size** | Columns taking disproportionate space | Remove if unused; shorten string values |
| **Dictionary Size** | Large dictionaries indicate high cardinality or long strings | Reduce cardinality; trim/shorten strings |
| **Relationship Size** | Large relationships may indicate modeling issues | Check for referential integrity violations |
| **Referential Integrity Violations** | Orphan keys in fact tables | Clean in Power Query or fix source data |
| **Total Model Size** | Overall memory footprint | Target: well within capacity limits |

### Common Findings and Fixes

1. **Remove unused columns**: Especially high-cardinality text columns in fact tables (descriptions, GUIDs, comments)
2. **Remove auto date/time tables**: If a custom date table exists, disable auto date/time in options
3. **Split high-cardinality columns**: Separate datetime into date + time columns (date has much lower cardinality)
4. **Address referential integrity violations**: Orphan keys create blank dimension rows and waste memory
5. **Reduce string length**: Long category names or descriptions increase dictionary size
6. **Remove calculated columns**: Replace with measures where possible; calculated columns persist in memory

## Slow Report Diagnosis

### Step-by-Step Workflow

1. **Performance Analyzer** -- Identify the slowest visuals on the page
2. **Copy DAX query** -- Take the generated DAX from the slowest visual
3. **DAX Studio Server Timings** -- Run the query; check SE vs FE split
4. **VertiPaq Analyzer** -- Check model for bloat, high-cardinality columns
5. **Report design review** -- Count visuals, slicers, check for hidden visuals

### Common Causes and Fixes

| Cause | Symptom | Fix |
|---|---|---|
| Too many visuals per page | All visuals slow; high total page load time | Reduce to 8 or fewer; use drillthrough pages |
| Complex DAX measures | High FE time in DAX Studio | Use variables, avoid nested iterators, use native aggregations |
| High-cardinality columns | Large model size in VertiPaq Analyzer | Remove or bucket unnecessary columns |
| Excessive slicers | Slow initial load and every interaction | Consolidate slicers; use filter pane |
| Bidirectional relationships | Unpredictable slow queries | Switch to single-direction; use DAX CROSSFILTER per-measure |
| Large Import tables | High memory, slow refresh | Implement incremental refresh; use aggregations |
| Missing relationships | Incorrect or slow results | Verify star schema with proper relationships |
| Many-to-many without bridge | Excessive cross-join results | Add proper bridge table |

## Data Refresh Failure Diagnosis

### Step 1: Check Refresh History

Navigate to semantic model > Settings > Refresh > Refresh History. Review status, duration, and error messages.

### Step 2: Categorize the Error

| Error Category | Common Messages | Resolution |
|---|---|---|
| **Credential errors** | "Credential error", "OAuth token expired" | Re-enter credentials in Data source credentials; for long refreshes, optimize to stay under 1-hour OAuth token lifetime |
| **Connectivity errors** | "Gateway not reachable", "Data source not found" | Restart gateway service; update to latest version; verify network/firewall |
| **Data errors** | "Column not found", "Type mismatch" | Source schema changed; update Power Query to match new schema |
| **Privacy errors** | "Information needed to combine data" | Set privacy levels explicitly in data source settings |
| **Memory errors** | "Exceeded capacity memory limit" | Reduce model size; upgrade capacity; implement incremental refresh |
| **Timeout errors** | "Query timeout" | Optimize Power Query; implement query folding; use incremental refresh |

### Step 3: Test in Desktop

Open the .pbix on the gateway machine; attempt refresh; check Power Query steps for errors. This isolates whether the issue is data source connectivity, query logic, or Service-specific.

### Step 4: Review Gateway Logs (if applicable)

Location: `%localappdata%\Microsoft\On-premises data gateway\`
Look for connection failures, authentication errors, timeout messages.

### Prevention

- Set up email alerts for refresh failures (built-in)
- After 4 consecutive scheduled refresh failures, the schedule is automatically deactivated
- Service principal authentication avoids OAuth token expiry issues for long-running refreshes
- Force credential refresh: `https://app.powerbi.com?alwaysPromptForContentProviderCreds=true`

### Refresh Types

| Type | Trigger | Max Frequency | Monitoring |
|---|---|---|---|
| Scheduled | Time-based | 8/day (Pro), 48/day (PPU) | Refresh history |
| On-Demand | Manual in Service | Unlimited | Refresh history |
| Incremental | Partition-based | Partition-level | XMLA endpoint + Refresh history |
| Real-Time (Hybrid) | DirectQuery partition | Continuous | DirectQuery diagnostics |
| API | REST API trigger | Unlimited | API response + Refresh history |

## Gateway Troubleshooting

### On-Premises Data Gateway

The gateway bridges on-premises data sources with the Power BI Service via outbound HTTPS connections.

### Troubleshooting Steps

1. **Verify gateway version**: Always update to latest; many issues resolved in monthly updates
2. **Check Windows service**: Ensure "On-premises data gateway" service is running
3. **Test local connectivity**: Open Power BI Desktop on the gateway machine and test the data source connection directly
4. **Review gateway logs**: `%localappdata%\Microsoft\On-premises data gateway\`
5. **Check firewall**: Gateway needs outbound HTTPS (port 443) to Azure Service Bus; no inbound ports required
6. **Verify credentials**: Re-enter data source credentials in Service > Manage gateways
7. **Test DNS resolution**: Ensure the gateway machine can resolve `*.servicebus.windows.net`

### Gateway Cluster Considerations

- Default concurrency: 30 concurrent queries per gateway node
- Add gateway nodes to a cluster for higher throughput and failover
- Load balancing distributes queries across nodes in a cluster
- Monitor CPU and memory on gateway machines; high utilization indicates need for additional nodes

### Personal vs Enterprise Gateway

| Feature | Personal | Enterprise (On-Premises) |
|---|---|---|
| DirectQuery support | No | Yes |
| Multi-user | No | Yes |
| Gateway cluster | No | Yes |
| Service principal auth | No | Yes |
| Production use | Not recommended | Yes |

### Fabric VNet Data Gateways

Managed gateway infrastructure eliminating the need to install and maintain gateway software. Available for Fabric capacity workspaces. Recommended for new deployments where possible.

## Capacity Management

### Fabric Capacity Metrics App

The primary tool for monitoring capacity health. Install from AppSource; connect to your Fabric capacity.

### Key Metrics

| Metric | Description | Action Threshold |
|---|---|---|
| **CU Utilization %** | CPU usage relative to capacity v-cores | Warning at 80%+ sustained during business hours |
| **Interactive Operations** | User-triggered queries | Monitor for throttling events |
| **Background Operations** | Scheduled refreshes, dataflow runs | Stagger to avoid peaks |
| **Throttling Events** | Operations delayed due to capacity saturation | Target: zero during business hours |
| **Overload Events** | Operations rejected due to sustained overload | Requires immediate action: reduce load or upgrade SKU |

### Throttling Behavior

When capacity is overloaded, throttling progresses through stages:
1. **Interactive throttling**: User queries are delayed (not rejected)
2. **Background throttling**: Scheduled refreshes are deferred
3. **Autoscale activation** (if configured): Additional v-cores added in 30-second increments
4. **Rejection**: Sustained overload leads to operation rejection

### When to Upgrade SKU vs Autoscale

Upgrade the base SKU when:
- Autoscale activates more than 3 times per week
- Sustained CU utilization above 80% during business hours
- Interactive throttling events occur regularly
- Autoscale activates daily (likely cheaper to upgrade)

Autoscale is a safety net for spikes, not a primary capacity strategy.

### Capacity Optimization Strategies

1. Stagger refresh schedules across 2-hour windows
2. Reduce semantic model sizes (remove unused columns/tables)
3. Implement incremental refresh for large datasets
4. Limit visuals to 8-12 per page
5. Use aggregations for common query patterns
6. Separate dev/test workloads onto different capacities
7. Weekly review of Capacity Metrics App utilization patterns

## DAX Debugging

### EVALUATE in DAX Studio

Test measures in isolation outside of reports:

```dax
-- Test a measure across dimensions
EVALUATE
SUMMARIZECOLUMNS(
    'Date'[Year],
    'Date'[Month],
    "Total Sales", [Total Sales],
    "YTD Sales", [YTD Sales]
)

-- Filter to specific context
EVALUATE
CALCULATETABLE(
    SUMMARIZECOLUMNS(
        'Product'[Category],
        "Revenue", [Revenue]
    ),
    'Date'[Year] = 2025
)
```

### Decompose Complex Measures

For complex measures, break into components using DEFINE VAR:

```dax
DEFINE
    VAR Revenue = [Total Revenue]
    VAR Cost = [Total Cost]
    VAR Margin = Revenue - Cost
    VAR MarginPct = DIVIDE(Margin, Revenue)
EVALUATE
ROW(
    "Revenue", Revenue,
    "Cost", Cost,
    "Margin", Margin,
    "Margin %", MarginPct
)
```

Examine each variable independently to find where the calculation diverges from expectation.

### Common DAX Debugging Scenarios

| Issue | Diagnostic Approach |
|---|---|
| Measure returns BLANK | Check filter context; verify relationships; test with EVALUATE and explicit filters |
| Measure returns wrong total | Check for context transition issues; test with ALL/REMOVEFILTERS to see unfiltered result |
| Slow measure | Server Timings in DAX Studio; check SE vs FE split; look for high materialization counts |
| Circular dependency error | Review calculated column/table dependencies; restructure to break cycles |
| Unexpected duplicates | Check relationship cardinality; verify key uniqueness in dimensions |
| RLS not filtering correctly | Test with "View as" role; run EVALUATE with USERPRINCIPALNAME() check |

## Monitoring and Alerting

### Built-in Monitoring

| Tool | Purpose | Access |
|---|---|---|
| **Refresh History** | Per-dataset refresh status and errors | Semantic model settings |
| **Usage Metrics** | Report views, unique viewers, performance | Report/workspace context menu |
| **Capacity Metrics App** | CU utilization, throttling, background ops | Fabric capacity admin |
| **Admin Portal** | Tenant-wide usage, audit logs, governance | Power BI admin role |
| **Activity Log** | Detailed audit events via REST API | Admin API or PowerShell |

### External Monitoring

| Tool | Purpose |
|---|---|
| **DAX Studio** | Query performance analysis, VertiPaq metrics |
| **Tabular Editor** | Best Practice Analyzer, model documentation |
| **Azure Monitor** | Capacity-level diagnostics and alerting |
| **Power BI REST API** | Programmatic monitoring and automation |
| **Power Automate** | Custom alerting on refresh failures or capacity events |

### Recommended Alerting Setup

1. Email subscriptions for refresh failures (built-in; enable immediately)
2. Power Automate flows for custom alerting (Teams, Slack, email)
3. Capacity Metrics App alerts for throttling/overload events
4. Azure Monitor for capacity diagnostics at scale
5. Scheduled Activity Log exports for compliance auditing
