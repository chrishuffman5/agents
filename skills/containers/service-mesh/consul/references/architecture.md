# Consul Connect Architecture

## Consul Server Cluster

### Raft Consensus

Consul servers form a Raft cluster (3 or 5 nodes recommended):

```
Consul Server Cluster:
  |-- Leader (handles all writes)
  |     |-- Service catalog mutations
  |     |-- KV store writes
  |     |-- Intention changes
  |     |-- ACL token creation
  |     +-- Raft log replication to followers
  |
  |-- Follower 1 (handles reads, receives Raft log)
  |-- Follower 2 (handles reads, receives Raft log)
  |
  |-- Consistency modes:
  |     default: leader handles reads (consistent, slightly slower)
  |     stale: any server handles reads (eventually consistent, faster)
  |     consistent: leader verifies leadership before read (strongest)
  |
  +-- Anti-entropy: periodically syncs state between servers and agents
```

### Server Architecture

```
Consul Server Process:
  |-- HTTP API (:8500)
  |     |-- Service registration/deregistration
  |     |-- Health check management
  |     |-- KV store operations
  |     |-- ACL management
  |     |-- Intention CRUD
  |     +-- Prepared queries
  |
  |-- gRPC (:8502)
  |     |-- xDS protocol (Envoy proxy configuration)
  |     |-- Consul Dataplane communication
  |     +-- Peering protocol
  |
  |-- DNS (:8600)
  |     |-- Service discovery via DNS
  |     |-- <service>.service.consul
  |     |-- <service>.service.<dc>.consul
  |     +-- SRV records for port information
  |
  |-- Serf (gossip protocol)
  |     |-- LAN gossip (:8301, UDP+TCP) -- within datacenter
  |     |-- WAN gossip (:8302, UDP+TCP) -- between datacenters
  |     |-- Membership management
  |     +-- Failure detection
  |
  +-- Raft (:8300)
        |-- Consensus protocol
        |-- Leader election
        |-- Log replication
        +-- Snapshot management
```

## Consul Dataplane Architecture

### Per-Service Sidecar Model

```
Pod / VM:
  +------------------------------------------------------------+
  |  Application Container                                      |
  |  (your service, unmodified)                                |
  |     |                                                       |
  |     | localhost traffic                                     |
  |     v                                                       |
  |  Consul Dataplane (lightweight Go process)                 |
  |  |-- Bootstraps Envoy proxy configuration                  |
  |  |-- Watches Consul servers for config changes (gRPC)      |
  |  |-- Translates Consul config to Envoy xDS                 |
  |  |-- Manages Envoy lifecycle (start, reload, drain)        |
  |  +-- Reports health back to Consul servers                 |
  |     |                                                       |
  |     v                                                       |
  |  Envoy Proxy (sidecar)                                     |
  |  |-- mTLS termination and origination                      |
  |  |-- Intention enforcement (L4/L7)                         |
  |  |-- L7 routing (ServiceRouter rules)                      |
  |  |-- Load balancing                                        |
  |  |-- Health checking (active and passive)                  |
  |  +-- Metrics emission (Prometheus)                          |
  +------------------------------------------------------------+
```

### Dataplane Communication Flow

```
1. Pod starts --> Consul Dataplane initializes
2. Dataplane connects to Consul server via gRPC (:8502)
3. Dataplane authenticates (ACL token)
4. Dataplane requests:
   - Service instance certificate (mTLS)
   - Upstream service endpoints
   - Intention rules
   - L7 routing configuration
5. Dataplane translates to Envoy xDS and boots Envoy
6. Envoy begins intercepting traffic
7. Dataplane watches for config changes (streaming gRPC)
8. On change: Dataplane pushes updated xDS to Envoy (hot reload)
```

## Connect CA (Certificate Authority)

### Built-in CA

```
Root CA (generated by Consul leader)
  |
  Intermediate CA (per datacenter)
  |
  Leaf Certificate (per service instance)
    Subject: spiffe://trust-domain/ns/namespace/dc/datacenter/svc/service
    Validity: 72 hours (default, configurable)
    Key: EC P-256
    Auto-rotation: at 50% of TTL
```

### Vault CA Integration

```hcl
# Consul server config for Vault CA
connect {
  ca_provider = "vault"
  ca_config {
    address = "https://vault.example.com:8200"
    root_pki_path = "connect-root"
    intermediate_pki_path = "connect-intermediate-dc1"
    token = "s.VaultToken"
    leaf_cert_ttl = "72h"
  }
}
```

Benefits of Vault CA:
- Centralized CA management across all Consul datacenters
- Hardware security module (HSM) support for root key
- Audit logging of all certificate operations
- Cross-datacenter certificate trust via shared Vault root

## Intention Evaluation

### L4 Intention Flow

```
New TCP connection:
  1. Client Envoy initiates mTLS to server Envoy
  2. Server Envoy extracts client identity from certificate
  3. Server Envoy evaluates intentions:
     - Find most specific matching intention (source + destination)
     - If ALLOW: accept connection
     - If DENY: reject connection (TCP reset)
  4. If no matching intention: default behavior (depends on ACL default policy)
```

### L7 Intention Flow

```
HTTP request (on established mTLS connection):
  1. Server Envoy receives HTTP request
  2. Evaluates L7 permissions:
     - Match request method, path, headers against permission rules
     - First matching permission wins (ordered evaluation)
  3. If ALLOW: proxy request to application
  4. If DENY: return 403 Forbidden
```

### Intention Caching

Intentions are cached in Envoy's configuration:
- Consul Dataplane pushes intention changes to Envoy via xDS
- Propagation time: typically < 5 seconds
- No restart required for intention changes

## L7 Traffic Management Chain

### ServiceRouter

Evaluates incoming requests and routes them based on HTTP attributes:

```
Incoming request
  |
  ServiceRouter
  |-- Match 1: pathPrefix=/api/v2 --> route to api-v2 service
  |-- Match 2: header x-canary=true --> route to api-canary service
  |-- Match 3: queryParam debug=true --> route to api-debug service
  +-- Default: route to api service (catch-all)
```

### ServiceSplitter

Distributes traffic by weight (after router selects the service):

```
Selected service: api
  |
  ServiceSplitter
  |-- 90% --> api (subset: v1)
  +-- 10% --> api (subset: v2)
```

### ServiceResolver

Selects specific instances and handles failover:

```
Selected service + subset: api v1
  |
  ServiceResolver
  |-- Filter: Service.Meta.version == v1
  |-- Connect timeout: 5s
  |-- Failover:
  |     v1 in dc1 unavailable --> try dc2 --> try dc3
  +-- Return selected endpoints to Envoy
```

### Full Chain Example

```
Request: GET /api/v2/users (x-canary: true)
  |
  ServiceRouter: pathPrefix=/api/v2 --> service: api-v2
  |
  ServiceSplitter: 90% v1, 10% v2
  |
  ServiceResolver: subset v2, filter version==v2, failover to dc2
  |
  Envoy: load balance across matched endpoints, mTLS
```

## Gateway Architecture

### Mesh Gateway

```
DC1:                                              DC2:
  Service A                                         Service B
    |                                                  ^
    v                                                  |
  Envoy sidecar (mTLS)                            Envoy sidecar (mTLS)
    |                                                  ^
    v                                                  |
  Mesh Gateway (Envoy, LoadBalancer)              Mesh Gateway (Envoy, LoadBalancer)
    |                                                  ^
    +-------------- WAN (mTLS over internet) ----------+
```

Mesh gateway routing:
- Inspects the SNI (Server Name Indication) of the mTLS connection
- Routes based on destination service name encoded in SNI
- No payload inspection required (L4 only)
- All traffic is encrypted end-to-end (double mTLS: sidecar + gateway)

### Mesh Gateway Modes

```yaml
# ServiceDefaults for mesh gateway mode
apiVersion: consul.hashicorp.com/v1alpha1
kind: ServiceDefaults
metadata:
  name: api
spec:
  meshGateway:
    mode: local    # default: route through local DC's gateway
```

| Mode | Behavior | Use Case |
|---|---|---|
| local | Traffic exits through local mesh gateway | Default, most common |
| remote | Traffic enters through remote mesh gateway | When remote DC controls ingress |
| none | Direct pod-to-pod (no gateway) | Flat network between DCs |

### Ingress Gateway

Exposes mesh services to external clients without a sidecar:

```yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: IngressGateway
metadata:
  name: ingress-gateway
spec:
  listeners:
  - port: 443
    protocol: http
    services:
    - name: frontend
      hosts: ["app.example.com"]
    - name: api
      hosts: ["api.example.com"]
  tls:
    enabled: true
    sds:
      clusterName: sds-cluster
      certResource: api-cert
```

### Terminating Gateway

Allows meshed services to reach external services with mTLS origination:

```
Meshed service --> Envoy sidecar --> Terminating Gateway --> External DB (TLS)
                                          |
                                     Terminates mesh mTLS
                                     Originates TLS to external service
```

## Transparent Proxy (Kubernetes)

### How It Works

1. `consul-connect-inject` webhook injects:
   - Consul Dataplane sidecar
   - Envoy proxy sidecar
   - init container that configures iptables
2. iptables rules redirect all outbound TCP to Envoy
3. Envoy resolves upstream via Consul DNS
4. mTLS is automatic for all meshed destinations
5. Non-mesh destinations pass through (unless explicitly denied)

### iptables Rules

```bash
# Outbound: redirect to Envoy outbound listener
iptables -t nat -A OUTPUT -p tcp -j REDIRECT --to-ports 15001

# Inbound: redirect to Envoy inbound listener
iptables -t nat -A PREROUTING -p tcp -j REDIRECT --to-ports 20000

# Exclude: Consul, DNS, and proxy traffic
iptables -t nat -A OUTPUT -d 127.0.0.1 -j RETURN
iptables -t nat -A OUTPUT -m owner --uid-owner envoy-uid -j RETURN
```

## Monitoring and Metrics

### Prometheus Metrics

Consul exposes metrics from multiple sources:

| Source | Metrics |
|---|---|
| Envoy sidecar | Request count, latency, error rate, connection pool |
| Consul Dataplane | xDS sync status, certificate rotation |
| Consul server | Raft commit time, KV operations, intention evaluations |
| Mesh gateway | Cross-DC request count, bandwidth |

### Key Metrics for Alerting

```yaml
# Alert: intention denials spike
- alert: ConsulIntentionDenials
  expr: rate(envoy_http_downstream_rq{consul_destination_service="api",envoy_response_code="403"}[5m]) > 10
  for: 5m

# Alert: mesh gateway latency
- alert: MeshGatewayLatency
  expr: histogram_quantile(0.99, rate(envoy_cluster_upstream_rq_time_bucket{consul_source_service="mesh-gateway"}[5m])) > 500
  for: 10m
```
